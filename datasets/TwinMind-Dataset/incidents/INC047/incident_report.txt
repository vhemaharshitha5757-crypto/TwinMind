1. SUMMARY
========================================================================

The API rate limiter (rate-limiter-prod-01) began incorrectly blocking legitimate API traffic at 13:15 UTC on December 1, 2025. The Redis-based rate counter experienced a clock drift issue after a Redis node restart, causing the sliding window counters to miscalculate request rates. Legitimate API consumers with normal traffic patterns were being rate-limited with 429 Too Many Requests responses.

This incident was classified as P2 severity and was assigned to
Jenny Liu in the API Engineering department. The affected system was
API Rate Limiter, specifically the rate-limiter-prod-01 component located in US-East-1 (AWS).
The incident was detected through automated monitoring and alerting systems,
and the response team was engaged within minutes of the initial alert.

========================================================================
2. BUSINESS IMPACT
========================================================================

25% of legitimate API traffic incorrectly rate-limited for 30 minutes. 35 API partners reported receiving 429 errors. Partner dashboards showed false degradation alerts. 3 partners escalated to account management. Trust in the API platform reliability was impacted.

The incident triggered the organization's major incident response process,
involving cross-functional coordination between engineering, operations, and
business stakeholders. Customer support was notified and prepared to handle
incoming queries related to the service degradation.

3. SYMPTOMS OBSERVED
========================================================================

The following symptoms were observed during the incident:

  * 429 Too Many Requests responses for API consumers well below their rate limits
  * Rate limiter Redis counters showing inflated request counts
  * Redis node clock showing 3-minute drift after restart
  * Sliding window algorithm producing incorrect calculations due to time skew
  * Only requests processed by the restarted Redis node affected

These symptoms were correlated across multiple monitoring systems including
application performance monitoring, infrastructure metrics, and log
aggregation platforms. The incident response team confirmed the symptoms
during their initial assessment and used them to guide the investigation.

5. INVESTIGATION

========================================================================
6. ROOT CAUSE ANALYSIS
========================================================================

The root cause of this incident was determined to be:
Redis node clock drift of 3 minutes after restart caused the sliding window rate limiting algorithm to miscalculate, counting requests from overlapping time windows and inflating rate counts.

Further analysis confirmed that this issue had been developing over time but
was not detected by existing monitoring thresholds and alerting rules. The
failure point was within the rate-limiter-prod-01 component of the API Rate Limiter
system. Contributing factors included insufficient monitoring coverage for
the specific failure mode encountered and the absence of automated
remediation procedures for this particular scenario.

========================================================================
7. RESOLUTION
========================================================================

The incident was resolved by the following actions: Synchronized Redis node clock via NTP, reset all rate limiting counters, and implemented time-independent rate limiting algorithm using token bucket instead of sliding window.

The resolution was executed by Jenny Liu with support from the incident
response team. After applying the remediation steps, the team monitored the
system for a stabilization period to confirm that the issue was fully
resolved and no secondary or cascading failures had occurred. All affected
services were verified to be operating within normal parameters before the
incident was formally closed.

========================================================================
8. PREVENTIVE ACTIONS
========================================================================

The following preventive actions have been identified to reduce the
likelihood and impact of similar incidents in the future:

  1. Implement NTP synchronization monitoring for all Redis nodes
  2. Migrate rate limiting from sliding window to token bucket algorithm
  3. Add rate limiter accuracy monitoring comparing actual vs measured rates
  4. Implement rate limiter bypass for known partners during clock drift events
  5. Create automated rate counter reset procedure for clock drift detection

These action items will be tracked in the engineering backlog and prioritized
based on risk assessment and business impact analysis.
