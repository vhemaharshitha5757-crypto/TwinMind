1. SUMMARY
========================================================================

CPU utilization on the search service ECS tasks spiked to 98% at 19:05 UTC on May 12, 2025. A recently deployed search query parser included an inefficient regular expression that caused catastrophic backtracking on certain user input patterns. The regex engine consumed all available CPU cycles when processing these specific queries, starving other request processing threads.

This incident was classified as P2 severity and was assigned to
Nina Patel in the SRE department. The affected system was
ECS Fargate Cluster, specifically the search-svc-task-08 component located in EU-West-1 (AWS).
The incident was detected through automated monitoring and alerting systems,
and the response team was engaged within minutes of the initial alert.

========================================================================
2. BUSINESS IMPACT

6. ROOT CAUSE ANALYSIS
========================================================================

The root cause of this incident was determined to be:
Catastrophic regex backtracking in search query parser caused by nested quantifiers in a recently deployed regular expression pattern.

Further analysis confirmed that this issue had been developing over time but
was not detected by existing monitoring thresholds and alerting rules. The
failure point was within the search-svc-task-08 component of the ECS Fargate Cluster
system. Contributing factors included insufficient monitoring coverage for
the specific failure mode encountered and the absence of automated
remediation procedures for this particular scenario.

========================================================================
7. RESOLUTION
========================================================================

The incident was resolved by the following actions: Rolled back the search service to the previous version without the problematic regex, deployed a fix with an optimized non-backtracking regex pattern, and added regex execution timeout.

The resolution was executed by Nina Patel with support from the incident
response team. After applying the remediation steps, the team monitored the
system for a stabilization period to confirm that the issue was fully
resolved and no secondary or cascading failures had occurred. All affected
services were verified to be operating within normal parameters before the
incident was formally closed.
