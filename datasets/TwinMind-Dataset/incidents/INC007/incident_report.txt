1. SUMMARY
========================================================================

Docker containers running the analytics batch processing workload on node analytics-worker-04 were repeatedly OOMKilled starting at 20:33 UTC on February 24, 2025. The container memory limit was set to 4GB but a recent code change introduced a data aggregation step that loaded entire result sets into memory instead of streaming them. Processing of large customer datasets triggered memory allocation beyond the container limit.

This incident was classified as P2 severity and was assigned to
Tom Williams in the Data Engineering department. The affected system was
Docker Swarm Cluster, specifically the analytics-worker-04 component located in EU-West-1 (AWS).
The incident was detected through automated monitoring and alerting systems,
and the response team was engaged within minutes of the initial alert.

========================================================================
2. BUSINESS IMPACT
========================================================================

Analytics batch processing pipeline stalled for 2.5 hours. Daily analytics reports for 350 enterprise customers were delayed. Real-time dashboard data became stale. No data loss occurred as the batch queue preserved unprocessed jobs. Business intelligence team unable to generate morning reports for EMEA leadership.

The incident triggered the organization's major incident response process,
involving cross-functional coordination between engineering, operations, and
business stakeholders. Customer support was notified and prepared to handle
incoming queries related to the service degradation.

6. ROOT CAUSE ANALYSIS
========================================================================

The root cause of this incident was determined to be:
Recent code change in analytics aggregation module loading entire SQL result sets into memory instead of using cursor-based streaming for large datasets.

Further analysis confirmed that this issue had been developing over time but
was not detected by existing monitoring thresholds and alerting rules. The
failure point was within the analytics-worker-04 component of the Docker Swarm Cluster
system. Contributing factors included insufficient monitoring coverage for
the specific failure mode encountered and the absence of automated
remediation procedures for this particular scenario.

========================================================================
7. RESOLUTION
========================================================================

The incident was resolved by the following actions: Deployed code fix to implement cursor-based streaming for large datasets, increased container memory limit to 8GB as interim measure, and reprocessed stalled analytics jobs.

The resolution was executed by Tom Williams with support from the incident
response team. After applying the remediation steps, the team monitored the
system for a stabilization period to confirm that the issue was fully
resolved and no secondary or cascading failures had occurred. All affected
services were verified to be operating within normal parameters before the
incident was formally closed.

========================================================================
8. PREVENTIVE ACTIONS
========================================================================

The following preventive actions have been identified to reduce the
likelihood and impact of similar incidents in the future:

  1. Implement memory profiling in CI pipeline for data processing services
  2. Add container memory usage alerting at 75% of limit
  3. Mandate cursor-based streaming for all database queries returning over 10K rows
  4. Create load testing with production-scale datasets in staging
  5. Implement progressive memory limit increases with validation

These action items will be tracked in the engineering backlog and prioritized
based on risk assessment and business impact analysis.
