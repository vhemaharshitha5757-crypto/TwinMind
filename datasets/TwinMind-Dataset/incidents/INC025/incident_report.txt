1. SUMMARY
========================================================================

A HashiCorp Vault secrets engine misconfiguration at 22:15 UTC on June 30, 2025 caused production application services to read staging database credentials instead of production credentials. The issue was introduced during a Vault policy migration where the secrets path mapping was incorrectly updated. Production services connected to the staging database, causing both data integrity concerns and staging environment instability.

This incident was classified as P1 severity and was assigned to
David Kim in the Security Engineering department. The affected system was
HashiCorp Vault, specifically the vault-prod-cluster-01 component located in US-East-1 (AWS).
The incident was detected through automated monitoring and alerting systems,
and the response team was engaged within minutes of the initial alert.

========================================================================
2. BUSINESS IMPACT
========================================================================

Production services wrote approximately 2,300 records to the staging database over 18 minutes before detection. Staging database performance degraded under unexpected production traffic load. Data isolation between environments was breached. Security incident review was triggered. No customer data was exposed externally.

The incident triggered the organization's major incident response process,
involving cross-functional coordination between engineering, operations, and
business stakeholders. Customer support was notified and prepared to handle
incoming queries related to the service degradation.

6. ROOT CAUSE ANALYSIS
========================================================================

The root cause of this incident was determined to be:
Vault secrets path mapping error during policy migration pointed production applications to the staging secrets engine path instead of the production path.

Further analysis confirmed that this issue had been developing over time but
was not detected by existing monitoring thresholds and alerting rules. The
failure point was within the vault-prod-cluster-01 component of the HashiCorp Vault
system. Contributing factors included insufficient monitoring coverage for
the specific failure mode encountered and the absence of automated
remediation procedures for this particular scenario.

========================================================================
7. RESOLUTION
========================================================================

The incident was resolved by the following actions: Corrected Vault secrets path mapping, rotated all database credentials for both environments, removed the 2,300 erroneous records from staging database, and verified environment isolation.

The resolution was executed by David Kim with support from the incident
response team. After applying the remediation steps, the team monitored the
system for a stabilization period to confirm that the issue was fully
resolved and no secondary or cascading failures had occurred. All affected
services were verified to be operating within normal parameters before the
incident was formally closed.

========================================================================
8. PREVENTIVE ACTIONS
========================================================================

The following preventive actions have been identified to reduce the
likelihood and impact of similar incidents in the future:

  1. Implement Vault policy change validation with environment isolation checks
  2. Add secrets path verification in application startup health checks
  3. Create automated environment isolation testing for Vault configurations
  4. Implement Vault policy review requiring dual approval
  5. Add database hostname verification in application connection initialization

These action items will be tracked in the engineering backlog and prioritized
based on risk assessment and business impact analysis.
