1. SUMMARY
========================================================================

A firewall rule change applied to the Azure Network Security Group (nsg-prod-api-tier) at 12:30 UTC on March 17, 2025 inadvertently blocked inbound HTTPS traffic on port 443 from the API gateway subnet. The change was part of a security hardening initiative that tightened ingress rules, but the new rules did not include an allow rule for the API gateway's updated IP range after a recent subnet expansion.

This incident was classified as P2 severity and was assigned to
David Kim in the Security Engineering department. The affected system was
Azure Network Security Groups, specifically the nsg-prod-api-tier component located in East US (Azure).
The incident was detected through automated monitoring and alerting systems,
and the response team was engaged within minutes of the initial alert.

========================================================================
2. BUSINESS IMPACT
========================================================================

All API traffic from the gateway to backend services was blocked for 18 minutes. External API consumers received 502 Bad Gateway errors. Internal services using the API layer experienced failures. Approximately 4,200 API calls failed per minute during the incident.

The incident triggered the organization's major incident response process,
involving cross-functional coordination between engineering, operations, and
business stakeholders. Customer support was notified and prepared to handle
incoming queries related to the service degradation.

3. SYMPTOMS OBSERVED

The following symptoms were observed during the incident:

  * 502 Bad Gateway errors on all API endpoints served by Azure backend
  * API gateway unable to reach backend services on port 443
  * NSG flow logs showing DENY actions for API gateway source IPs

  * Only traffic from the expanded API gateway subnet range blocked

These symptoms were correlated across multiple monitoring systems including

8. PREVENTIVE ACTIONS
========================================================================

The following preventive actions have been identified to reduce the
likelihood and impact of similar incidents in the future:

  1. Implement pre-deployment NSG rule validation against known traffic patterns
  2. Create automated testing for NSG changes using synthetic API traffic
  3. Maintain a service dependency map linking NSG rules to traffic flows
  4. Require dual-approval for NSG rule modifications in production
  5. Add NSG flow log monitoring for unexpected DENY events

These action items will be tracked in the engineering backlog and prioritized
based on risk assessment and business impact analysis.
