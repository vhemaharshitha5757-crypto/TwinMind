1. SUMMARY
========================================================================

The User Service MongoDB connection pool was exhausted at 20:10 UTC on September 15, 2025. A new feature deployment introduced a code path that opened MongoDB connections without properly closing them in error cases. Under load, connection leaks accumulated until the pool reached its 500-connection maximum. New requests could not obtain database connections and failed with timeout errors.

This incident was classified as P2 severity and was assigned to
James Wright in the Backend Engineering department. The affected system was
MongoDB Atlas Cluster, specifically the mongo-user-prod-cluster component located in US-East-1 (AWS).
The incident was detected through automated monitoring and alerting systems,
and the response team was engaged within minutes of the initial alert.

========================================================================
2. BUSINESS IMPACT
========================================================================

User authentication and profile loading failed for 25 minutes. All user-related API endpoints returned 503 errors. Approximately 7,200 users unable to log in or access their profiles. Mobile app users experienced blank profile screens.

The incident triggered the organization's major incident response process,
involving cross-functional coordination between engineering, operations, and
business stakeholders. Customer support was notified and prepared to handle
incoming queries related to the service degradation.

Upon receiving the automated alert, the incident response team initiated a

========================================================================
6. ROOT CAUSE ANALYSIS
========================================================================

The root cause of this incident was determined to be:
Code path in new feature deployment opened MongoDB connections in a try block but failed to close them in the catch/error handler, causing connection leaks under error conditions.

Further analysis confirmed that this issue had been developing over time but
was not detected by existing monitoring thresholds and alerting rules. The
failure point was within the mongo-user-prod-cluster component of the MongoDB Atlas Cluster
system. Contributing factors included insufficient monitoring coverage for
the specific failure mode encountered and the absence of automated
remediation procedures for this particular scenario.

========================================================================
7. RESOLUTION
========================================================================

The incident was resolved by the following actions: Deployed hotfix to properly close connections in error handlers using try-with-resources pattern, restarted user service instances to release leaked connections, and verified pool recovery.

The resolution was executed by James Wright with support from the incident
response team. After applying the remediation steps, the team monitored the
system for a stabilization period to confirm that the issue was fully
resolved and no secondary or cascading failures had occurred. All affected
services were verified to be operating within normal parameters before the
incident was formally closed.

========================================================================
8. PREVENTIVE ACTIONS
========================================================================

The following preventive actions have been identified to reduce the
likelihood and impact of similar incidents in the future:

  1. Enforce try-with-resources pattern for all database connection usage via static analysis
  2. Add connection pool utilization monitoring with alerts at 70% capacity
  3. Implement connection pool leak detection in integration tests
  4. Set connection pool max lifetime to force recycling of stale connections
  5. Add connection leak detection logging in the MongoDB driver configuration

These action items will be tracked in the engineering backlog and prioritized
based on risk assessment and business impact analysis.
